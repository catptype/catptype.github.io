YOLO Inference Toolkit

I developed this open-source toolkit to solve a common problem for AI developers: the repetitive setup code and complexity of using the powerful Ultralytics YOLO framework. To simplify this, I built a set of intuitive, object-oriented Python classes that provide a clean and consistent workflow for every task. This lets developers get useful, ready-to-use data with a simple predict-then-decode process. The goal is to help developers move from idea to implementation much faster by focusing on their application, not the framework's internal details.

Project Goals & Design Philosophy:

  • Simplify Developer Experience: Abstract away complex framework internals and provide a simple, intuitive API for common AI tasks.
  • Promote Code Reusability: Use an object-oriented inheritance structure to prevent code duplication and make the toolkit easy to extend.
  • Provide Predictable Data: Ensure all decode methods return simple, clean Python data structures (lists, tuples), not complex objects.
  • Ensure Versatility: Design a model-agnostic tool that works seamlessly for classification, detection, segmentation, and pose estimation.

Key Features & Architecture:

  • Model-Agnostic Design: Works with any model type supported by the Ultralytics framework, including classification, detection, segmentation, and pose estimation.
  • Simplified Data Output: All decode methods are designed to return simple Python data structures (lists and tuples), not complex framework objects, making the results immediately usable.
  • Consistent "Predict → Decode" Pattern: The entire inference process is streamlined into two clear steps that work identically for both single images and batches.
  • Specialized Task Capabilities: Includes ready-to-use methods for advanced tasks like stateful object tracking and utilities to easily extract detected or segmented objects from an image.

Blue Archive Gacha Simulator

As a fan of the game Blue Archive, I built this feature-rich web application to challenge and showcase my full-stack skills. This is a non-profit, fan-made project created for fun and educational purposes, simulating the gacha and character collection experience using Django for the backend and Tailwind CSS for a responsive, modern frontend.

Core Features & Implementation:

  • Realistic Gacha Simulation: Implemented a secure backend service to handle pulls with banner-specific rates, complete with high-impact animations for revealing rare characters.
  • Data Visualization with Chart.js: Integrated interactive doughnut and bar charts to provide users with a visual breakdown of their gacha history and per-banner statistics.
  • Full Achievement & Progression System: Designed a complete backend service that awards achievements in real-time for collection milestones and feats of luck, with non-intrusive "toast" notifications.

Architecture & Design Patterns:

  • Asynchronous Dashboard: Developed a dynamic, widget-based dashboard that uses asynchronous JavaScript (Fetch API) to load all data widgets independently, preventing page reloads and improving user experience.
  • Optimized Performance with Caching: Implemented a caching layer (using the cache-aside pattern) to intercept database requests. This system serves frequently accessed data from a high-speed cache, significantly reducing redundant queries and improving application responsiveness.
  • Stateless Service Architecture: Decoupled core logic (Gacha Engine, Achievement Service) into stateless service classes, promoting separation of concerns and making the codebase cleaner and more testable.
  • Zero-Downtime Content Management: The application is architected for live content updates, fetching all core data dynamically from the database. This design allows new content to appear on the site instantly after a database update, requiring no server restarts or redeployment.
  • Dual User Modes: Features a full user authentication system (powered by Django) to save all gacha history, achievements, and collections, while also supporting a guest mode for instant, no-registration access to the core simulator.
  • Production vs. Development Environments: The application is configured to use PostgreSQL in production for robustness (on Render) while defaulting to SQLite for easy local development.
